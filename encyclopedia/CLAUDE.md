# Lexicon

Multi-project React app — personal English vocabulary encyclopedia with bilingual support (RU/EN) and wiki-style cross-linking between entries.

## Tech stack

- React 19 + TypeScript 5.7 (strict mode)
- Vite 6 (dev server + build + API middleware)
- TanStack Router (code-based routing, 3 routes)
- TanStack Query (data fetching)
- SQLite via `better-sqlite3` (data store)
- Playwright (e2e tests)
- PM2 (process manager)

## Commands

```bash
yarn dev              # Vite dev server (default port 5173)
yarn build            # tsc + vite build → dist/
yarn test             # Playwright e2e tests
yarn pm2              # Start via PM2 on port 4173
yarn pm2:restart      # Restart PM2 process
yarn pm2:stop         # Stop PM2 process
yarn db:import        # One-shot: import JSON snapshots into SQLite
yarn db:export        # Export SQLite back to JSON (for ghpages static build)
```

## Data store

Runtime store is `data/lexicon.db` (SQLite, WAL). Tables:

- `projects(id, name, title, subtitle, created_at)`
- `words(id, project_id, word, pronunciation, part_of_speech, cefr_level, forms, image, audio, meta)` — language-neutral fields
- `word_translations(word_id, lang, definitions, main_examples, usage_note, comparisons, collocations, idioms, related_forms, word_history, context_story)` — per-language
- `word_links(project_id, source_word_id, lang, target_word_lc, target_word_id, field, display_text)` — materialized wiki graph

Schema migrations live in `src/server/migrations/NNN_*.sql` and run automatically on first DB open.

**The DB file itself is NOT committed** — it's a large binary that can't be diffed or merged. The committed source of truth is the per-project JSON snapshots in `public/data/projects/<id>/{project,words.en,words.ru}.json`. These are both the ghpages data source AND the canonical content:

- After a fresh clone, run `yarn db:import` to rebuild `data/lexicon.db` from the JSON snapshots.
- After editing content (via `./scripts/lexicon.sh` or the API), run `yarn db:export` before committing — the DB change is only persisted in the snapshots.
- `data/*.db`, `-wal`, `-shm` are all gitignored.

## Wiki cross-linking

Write `[[target]]` or `[[target|display]]` inside any text field. The parser runs in `formatText` before markdown formatting and emits:
- `<a class="wiki-link" href="#<slug>">` if target exists in the project
- `<span class="wiki-link link-missing">` if broken

Targets resolve against the same project (cross-project syntax `[[project:word]]` is reserved). Forward references are supported — links are stored with `target_word_id = NULL` until the target is later added, at which point `upsertWord` resolves them automatically.

Links are materialized into `word_links` on every `upsertWord` call. Each card shows a "Used in N articles" backlinks section when it has incoming links.

## Key conventions

- Language preference persisted in `localStorage` (key: `lexicon-lang-{projectId}`)
- Words sorted alphabetically (`COLLATE NOCASE` in SQL)
- Sidebar tracks active word via scroll position
- Hash navigation (`#slug`) works for wiki-link clicks — `hashchange` listener scrolls virtualizer
- All styles live in `App.css`
- Fonts: "Source Serif 4" (body), "Inter" (UI) — loaded from Google Fonts

## Adding words

```bash
./scripts/lexicon.sh --project tainted-grail add < word.json              # Add to RU (default)
./scripts/lexicon.sh --project tainted-grail --lang en add < word.json    # Add to EN
./scripts/lexicon.sh --project tainted-grail list                          # List all word names
./scripts/lexicon.sh --project tainted-grail get "abhor"                   # Get single word entry
./scripts/lexicon.sh --project tainted-grail last 5                        # Last 5 added
./scripts/lexicon.sh --project tainted-grail backlinks "abhor"             # Words linking to abhor
./scripts/lexicon.sh --project tainted-grail broken                        # Broken wiki links
./scripts/lexicon.sh --project tainted-grail auto-linkify-compare          # Wrap comparisons[].word mentions in description with [[...]]
```

The shell script is a thin shim over `scripts/lexicon.mjs` (Node + better-sqlite3).

## Image generation

Uses **Cloudflare Workers AI** with `@cf/bytedance/stable-diffusion-xl-lightning` (beta; `cf-ai-neurons: 0.00` — doesn't count against the 10 000 neurons/day free tier, effectively unlimited). 1024×1024 JPEG, ~3–5 s per image.

```bash
./scripts/generate-image.sh --project tainted-grail "<word>" "<prompt>"
```

Credentials live in `scripts/.cloudflare.json` (gitignored):

```json
{
  "apiToken": "cfut_…",
  "accountId": "…",
  "model": "@cf/bytedance/stable-diffusion-xl-lightning",
  "numSteps": 8,
  "width": 1024,
  "height": 1024,
  "guidance": 0
}
```

The token needs only `Workers AI: Read` permission. Images land in `public/data/projects/<id>/images/words/<word>.jpg`. Path goes into `words.image` via `set-field` or full `add`.

### Prompt-engineering rules for SDXL-Lightning

Lightning is distilled and trained **without classifier-free guidance** — `guidance=0` is mandatory, otherwise composition degrades sharply. Two practical consequences for prompts:

- **One dominant subject.** Lightning cannot hold two opposing aesthetic poles in one scene — describing «small ugly thing next to huge beautiful thing» ends up merged into one mediocre hybrid or loses one side. Always craft scenes around a single primary subject with supporting context.
- **Avoid text-rendering triggers.** Words like `advertisements, graffiti, lettering, text, letters, scrawled, signs, labels, words, logos` activate SDXL's broken text mode and produce gibberish. Use visual decay markers instead (`rusted, crumbling, peeling, broken, bent, moss-covered, weather-beaten`).

Style anchor that works reliably: *«Beautiful painterly illustration style, vibrant saturated colors, cinematic composition, atmospheric lighting. IMPORTANT: absolutely no text, no letters, no words anywhere in the image.»*

Full prompt recipe and word-category templates live in `.claude/skills/generate-word-image/SKILL.md`.

## Pronunciation audio

Words can have an `audio` field storing external CDN URLs for pronunciation playback:

```typescript
audio?: { us?: string; uk?: string; au?: string }
```

```bash
./scripts/fetch-audio-urls.sh "hello"      # Prints JSON: {"us":"https://...","uk":"https://..."}
```

Queries Free Dictionary API (primary) and Wiktionary/Wikimedia Commons (fallback). URLs point to external CDNs. The `audio` field is language-neutral (stored once in `words.audio`).

## Memory & auto-learning (claude-flow hooks)

Persistent memory is wired via `.claude/helpers/hook-handler.cjs` (ADR-048 Phase 2). Every Edit/Write and task completion dual-writes:

1. **MCP memory** (`sql.js + HNSW`) — keyed entry under a typed namespace, visible via `mcp__claude-flow__memory_*`.
2. **AutoMemoryBridge JSON store** (`.claude-flow/data/auto-memory-store.json`) — picked up on session end by `auto-memory-hook.mjs sync`, projected into `~/.claude/projects/<proj>/memory/MEMORY.md` + topic files.

### Namespace convention

| Namespace | Writer | Content |
|-----------|--------|---------|
| `tasks` | `post-task` hook | Task summaries (`[agent] prompt`), confidence 0.75 |
| `edits` | `post-edit` hook | File edit events (`Edited <path> via <tool>`), confidence 0.65 |
| `learnings` | Manual / bridge | Curated insights tagged `insight` — only these reach `MEMORY.md` |

Only entries in `learnings` with `tags: ['insight']` and `confidence >= 0.7` are synced to `MEMORY.md`. The hook handler automatically mirrors every MCP write as a `learnings` insight in the JSON store for bridge consumption.

### Manual memory ops

```bash
claude-flow memory list                    # List all entries
claude-flow memory search -q "wiki link"   # Semantic search (HNSW)
claude-flow memory stats                   # Namespace counts, embedding coverage
node .claude/helpers/auto-memory-hook.mjs sync   # Force flush to MEMORY.md
node .claude/helpers/auto-memory-hook.mjs status # Bridge state + entry count
```
